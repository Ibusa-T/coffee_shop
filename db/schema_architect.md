# 3層スキーマ構造と設計フェーズの対応まとめ

データベース設計における「設計フェーズ」と、システム構造としての「3層スキーマ」の関係性は以下の通りです。

## 1. 対応マップ

| 3層スキーマの名称 | 該当する設計フェーズ | 主な内容・役割 |
| :--- | :--- | :--- |
| **外部スキーマ (External Schema)** | ビュー設計 / UI設計 |
| **「どう見えるか」**: ユーザーやアプリ、分析者が直接触れる部分（Viewなど）。 |
| **概念スキーマ (Conceptual Schema)** | 概念設計 / 論理設計 |
| **「何があるか / どう繋がるか」**: DB全体の論理的な実体。正規化されたテーブル群。| 
| **内部スキーマ (Internal Schema)** | 物理設計 | 
| **「どう格納するか」**: 特定のDB（PostgreSQL等）に依存したデータの持ち方、型、性能。 |

## 2. 各フェーズの詳細解説

### ① 概念設計 ➡ 概念スキーマの種
- **やること**: ビジネス要件からエンティティ（顧客、商品など）を抽出し、その関係性を定義する。
- **成果物**: ER図（手書きレベル）、エンティティリスト。

### ② 論理設計 ➡ 概念スキーマの完成
- **やること**: 特定のDB製品は意識せず、データの重複を排除（正規化）し、誰が見ても矛盾のない構造を作る。
- **成果物**: テーブル定義、主キー(PK)、外部キー(FK)の確定。
- **ポイント**: ここをしっかり作ることで、「論理データ独立性」（内部の格納方法が変わっても外部への見せ方を変えなくて済む性質）が確保されます。

### ③ 物理設計 ➡ 内部スキーマ
- **やること**: PostgreSQLなどの特定のDB製品を対象に、実際のパフォーマンスやデータ型を定義する。
- **成果物**: `CREATE TABLE`文（DDL）、データ型（`INT`, `VARCHAR`等）、インデックス、パーティショニング。
- **ポイント**: 「物理データ独立性」（データの持ち方を変えても、テーブル間の論理構造に影響を与えない性質）を意識します。

## 3. なぜこの分離が大事なのか？（エンジニアの視点）

この3つの層を切り分けて考えることで、以下のメリットが生まれます。

- **変更に強い**:
  - 例：売上の集計ロジックが変わっても（外部）、テーブル構造（概念）やデータの保存形式（内部）をいじらず、ViewのSQLを書き換えるだけで対応できる。
- **性能改善がしやすい**:
  - 例：検索が遅くなった時、テーブル構成（概念）は変えずに、インデックスの追加やストレージ設定の変更（内部）だけで解決できる。
- **役割分担の明確化**:
  - ビジネスルール担当（概念）と、パフォーマンス・チューニング担当（内部）が衝突せずに作業できる。
